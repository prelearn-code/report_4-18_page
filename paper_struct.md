下面我按**前端可直接落地**的方式，把这篇方案整理成一套完整流程。论文本身没有定义 REST / RPC API，所以我把它抽象成三类动作：**客户端本地计算**、**CSP 的链下接口**、**区块链/智能合约的链上接口**。整体系统和总工作流在图 1；上传去重在第 6–7 页，审计在第 7–8 页，检索/更新/交易在第 8–9 页。

## 1. 参与方与交互通道

有 3 个角色：

1. **用户端 U**
   负责本地分块、加密、生成 tag / authenticator、发起审计、更新、检索、交易。

2. **CSP**
   负责链下存储、去重检查、证明生成、MHT 维护、所有权记录更新。

3. **Blockchain + Smart Contract**
   负责存证、自动发 challenge、验证 proof、收付款、保证金惩罚、记录交易结果。

所以前端至少要同时连两类后端：

* **链下服务**：CSP API
* **链上服务**：合约 + 钱包

---

## 2. 前端要管理的核心数据

前端/客户端至少要有这些状态对象：

* **文件级**

  * 原始文件 `F`
  * 文件标签 `t`
* **块级**

  * 块标签集合 `T={tg_i}`
  * 唯一块集合 `C_u`
  * 认证器集合 `σ={σ_i}`
  * 密钥密文 `{Key_i}`
* **身份级**

  * `uid`
  * 随机数 `μ`
  * `UID`
  * `W`
* **本地私钥**

  * `sk`（ECC 私钥）
* **链上状态**

  * 存储费 `fee1`
  * 审计费 `fee2`
  * 交易费 `fee3`
  * CSP 保证金 `Fee1=2*fee1`
  * 卖家保证金 `Fee3=2*fee3`
  * MHT 根 `wr`
  * challenge `Chal`
  * proof `P`
  * audit result `result`

这个很关键：**如果你按论文真的做前端，本地不能只存 `t, sk`**。因为后续检索、更新、交易都还要提交 `UID, W`。论文上传末尾写“保留 `<t,sk>`”，但后续流程又要求带 `UID/W`，所以工程上你至少还要额外保存：

* `uid + μ`，后续现算 `UID/W`
  或
* 直接保存 `UID/W`。
  这是前端实现时必须补上的。

## 3. 完整主流程

# 3.1 系统初始化 / 页面进入

这一步对应论文里的 `Setup(λ)`，通常不是普通用户每次手动做，而是系统部署后前端初始化时读取：

* 公共参数 `para`
* 合约地址
* 当前用户钱包地址/账户
* 用户身份 `uid`
* CSP 列表或当前选中的 CSP

前端启动后建议先做：

1. 连接钱包
2. 拉取合约状态
3. 拉取用户在 CSP 的文件列表 / 交易列表 / 审计记录。

---

# 3.2 文件上传 + 用户侧去重

这是最核心的一条链路。

### Step A：用户选文件

前端上传页：

1. 选择文件 `F`
2. 选择 CSP
3. 标记哪些内容/块是**公开块** `F1`，哪些是**私密块** `F2`

论文假设文件会被拆成公开块和私密块，但**没有规定这个划分规则**，所以前端必须自己设计：

* 按目录/文件类型标记
* 按块范围标记
* 或者默认整文件私密/部分公开。

### Step B：本地预处理

客户端本地做：

1. 把文件分成 `n` 个 block
2. 每个 block 再分成 `s` 个 sector
3. 计算文件密钥
   `fk = H1(F)`
4. 计算每个 sector 的 key
   `k(i,j) = H1(m(i,j))`
5. 对私密块加密，对公开块不处理

   * 公开块直接保留明文
   * 私密块做 sector 级加密，得到 `c(i,j)`
6. 拼出处理后的数据集 `C`

   * 公开部分是明文块
   * 私密部分是密文块
7. 计算标签

   * 文件 tag：`t = g^fk`
   * 块 tag：`tg_i = H1(c_i)`
8. 生成 ECC 密钥对

   * 私钥 `sk`
   * 公钥 `pk`
9. 用 `pk` 把每个块对应的 sector key 打包加密，得到 `{Key_i}`
10. 生成身份凭证

* 选随机数 `μ`
* `UID = (H4(uid) · t)^μ`
* `W = g^μ`。

### Step C：链下去重预检查

前端向 CSP 发第 1 次请求，建议抽象成：

`POST /dedup/check`

请求体大致是：

* `t`
* `T={tg_i}`
* `{Key_i}`
* `uid`
* `UID`
* `W`

CSP 做两件事：

1. 验证用户身份是否合法
   `e(UID,g) = e(H4(uid)·t, W)`
2. 去重检查

   * 先做**文件级去重**：查 `t`
   * 如果整文件已存在，直接终止上传
   * 如果整文件不存在，再做**块级去重**：查 `T`
   * 返回唯一块标签集合 `T_u`。

### Step D：客户端只为唯一块生成认证器

前端收到 `T_u` 后，本地只对**唯一块**做后续操作：

1. 找到唯一块集合 `C_u`
2. 对每个唯一块计算

   * `y_i = (Σ k(i,j)) XOR γ`
   * `Y_i = g^(y_i)`
   * `σ_i = (H2(s||tg_i) · Π r_j^(c(i,j)))^(y_i)`

然后前端准备第二次请求。

### Step E：上传唯一块 + 认证器

前端向 CSP 发第 2 次请求，建议抽象成：

`POST /upload/commit`

请求体：

* `C_u`
* `σ={σ_i}`

同时，前端调用链上合约：

* 支付存储费 `fee1`

### Step F：CSP 存储落库 + 链上登记

CSP 收到后：

1. 验证块和认证器是否一致
   用论文公式 (5)
2. 验证通过则存储

   * `<t, T, {Key_i}, T_u, C_u>`
3. 构建增强版 MHT

   * 叶子存 `tg_i`
   * 内部节点哈希聚合
   * 得到根 `wr`
4. CSP 向链上存保证金
   `Fee1 = 2*fee1`
5. CSP 把存储信息上链：

   * `<t, T_u, σ, uid, SID, wr>`

最后，客户端本地删除大文件，只保留最少必要元数据。工程上建议保留：

* `t`
* `sk`
* `uid`
* `μ` 或 `UID/W`
* 文件名、CSP、链上 tx hash、当前 `wr`。

# 3.3 混合审计流程

这个流程是“明文块 + 密文块”统一审计。

### Step A：用户发起审计

前端调用链上合约：

* 提交 `req`
* 支付审计费 `fee2`

### Step B：合约自动生成 challenge

合约生成：

* `Chal = (z, θ1, θ2)`

然后把 `Chal` 发布到链上。

### Step C：CSP 读取 challenge 并生成 proof

CSP 监听链上 challenge 后，计算：

* `x_i = f1(i, θ1)`
* `v_i = f2(i, θ2)`
* 每个 sector 的证明：
  `P_j = Σ v_i * c(x_i,j)`

最终得到 proof：

* `P = {P_j}`

然后提交到链上。

### Step D：合约验证 proof

合约侧先聚合认证器：

* `σ_c = Π σ_(x_i)^(v_i)`

再按公式 (8) 校验 proof 是否通过。

### Step E：记录结果 + 结算

合约把结果上链：

* `<σ_c, P, result>`

结算规则：

* `result = 1`：说明审计通过，审计费 `fee2` 给 CSP
* `result = 0`：说明数据有问题，CSP 的保证金 `Fee1` 赔给用户。

前端这里最好做成一个**时间线**：

1. 审计请求已提交
2. challenge 已生成
3. proof 已提交
4. verify 完成
5. 资金已结算

# 3.4 数据检索 / 下载恢复

### Step A：用户发起检索

前端向 CSP 发请求，建议抽象成：

`POST /data/retrieve`

请求体：

* `t`
* `UID`
* `W`

### Step B：CSP 验证身份

还是验证：
`e(UID,g) = e(H4(uid)·t, W)`

通过后返回：

* `C`
* `{Key_i}`。

### Step C：客户端本地恢复文件

前端/客户端本地做：

1. 用 `sk` 解密 `Key_i`
   得到 sector key
2. 对私密块逐 sector 解密：
   `m(i,j) = H3(k(i,j)) XOR c(i,j)`
3. 公开块直接读取
4. 合并全部块，恢复原文件

所以检索页本质上要展示 3 类状态：

* 身份验证中
* 数据下载中
* 本地解密恢复中。

# 3.5 动态更新流程

论文明确支持：

* 插入 `Insert`
* 修改 `Modify`
* 删除 `Delete`

其中**插入写得最详细**，修改和删除只说“类似插入”，细节省略了。前端可按同一套路实现。

## 3.5.1 插入新块

### 第 1 阶段：初始化插入请求

用户要在第 `i` 个位置后插入新块 `m*`。

客户端先本地处理新块：

1. 如果是私密块：

   * 算 `k*_j = H1(m*_j)`
   * 加密得到 `c*`
2. 如果是公开块：

   * 直接 `c* = m*`
3. 计算 `tg* = H1(c*)`

然后前端先发**初始插入请求**给 CSP：

`POST /update/insert/init`

请求体：

* `Insert`
* `UID`
* `W`
* `t`
* `i`
* `tg*`

CSP 做：

* 身份验证
* 去重检查。

### 第 2 阶段：提交真正的数据块

如果 CSP 判断这个新块是唯一块，客户端再计算：

* `y* = (Σ k*_j) XOR γ`
* `Y* = g^(y*)`
* `σ* = (H2(s||tg*) · Π r_j^(c*_j))^(y*)`

然后发第二次请求：

`POST /update/insert/commit`

请求体：

* `Insert`
* `c*`
* `σ*`

### 第 3 阶段：CSP 更新存储和 MHT

CSP 做：

1. 存新块
2. 在 MHT 中插入叶子
3. 重新计算根 `wr*`
4. 上链更新信息：

   * `<t*, tg*, σ*, uid, SID, wr*>`

合约验证新根后，更新完成。图 2 就是这个增强版 MHT 的更新思路。

---

## 3.5.2 修改块

论文没把步骤展开，但可以按插入流程同构实现：

### 建议前端流程

1. 用户选择第 `i` 块
2. 本地生成新块 `c*`、`tg*`
3. 发初始化请求：

   * `Modify, UID, W, t, i, tg*`
4. CSP 验证身份 + 去重
5. 若需上传新块，则客户端生成 `σ*`
6. 提交：

   * `Modify, c*, σ*`
7. CSP 替换第 `i` 块
8. 更新 MHT，得到 `wr*`
9. 上链记录新状态。

---

## 3.5.3 删除块

论文也没展开，但按 `Delete(Delete,t,i)->wr*` 和“类似插入”可以这么落：

### 建议前端流程

1. 用户选择删除第 `i` 块
2. 发请求：

   * `Delete, UID, W, t, i`
3. CSP 验证身份
4. CSP 删除块或标记叶子不可访问
5. 更新 MHT，得到 `wr*`
6. 把新根和更新记录上链

删除通常不需要上传新块内容，也不需要新认证器。

# 3.6 数据所有权交易流程

这条链路在图 3 里最清楚。

## Step A：买家发起交易

买家 `U2` 在前端点击“购买/接收所有权”，链上提交：

* `<uid1, uid2>`
* 支付交易费 `fee3`

## Step B：卖家确认并交保证金

卖家 `U1` 提交：

* `<UID1, W1, t>`
* 保证金 `Fee3 = 2*fee3`

## Step C：合约通知 CSP 改所有权

智能合约把以下信息发给 CSP：

* `<UID1, W1, t, uid1, uid2>`

CSP 验证卖家身份后：

1. 更新云端所有权记录，把 owner 从 `uid1` 改成 `uid2`
2. 返回给合约：

   * `<t, T_u, uid2, SID, wr>`

链上记录这次所有权变更。

## Step D：卖家把解密私钥发给买家

所有权记录更新后：

* `U1` 通过安全信道把 `sk` 发给 `U2`
* 合约同时把 `fee3` 转给 `U1`

注意：这里**交易费先给卖家，保证金留在合约里兜底**。

## Step E：买家验证是否真的拿到了正确密钥

买家本地生成：

* `UID2 = (H4(uid2)·t)^μ2`
* `W2 = g^μ2`

然后向 CSP 发检索请求：

* `<UID2, W2, t>`

CSP 验证新 owner 身份，返回：

* `C`
* `Key_i`

买家尝试用收到的 `sk` 解密。

## Step F：交易成功/失败结算

* **解密成功**

  * `U2` 向合约发成功通知
  * 合约退还 `Fee3` 给 `U1`
* **解密失败**

  * `U2` 向合约发失败通知
  * 合约把 `Fee3` 赔给 `U2`

因为 tag 和 authenticator 只和数据内容有关，不和 owner 绑定，所以**所有权转移后 CSP 不需要重算 tag / authenticator**。这对前端也很重要：交易页只需要处理身份和支付，不需要重新上传数据。

## 5. 你做前端时必须提前补齐的 4 个“协议空白”

1. **本地到底保存什么**
   论文写只留 `<t,sk>`，但后续还要 `UID/W`。
   工程上建议最少保留：`t, sk, uid, μ`。

2. **公开块/私密块如何划分**
   论文没给 UI 规则，前端要自己设计。

3. **更新后的 `t*` 如何定义**
   论文更新时写了 `t*`，但没把重算过程展开。
   前端实现时，建议把“最新文件 tag / root”作为后端或合约返回值，不要前端自己猜。

4. **更新时如果新块是重复块怎么办**
   论文写了“如果唯一则上传”，但没详细展开“重复块插入/修改”的分支。
   这个分支要在后端协议里补清楚。
